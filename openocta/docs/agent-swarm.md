# Agent Swarm（蜂群协作）产品说明

## 一、产品定位

**Agent Swarm** 是 OpenOcta 中的多 Agent 协作能力：用户在一个**房间**里发起复杂任务，由根助手按需拆分子 Agent，形成树状协作关系；同时可以像使用微信群聊一样，随时向任意成员发消息、查看流式回复，并在实时拓扑图上观察整体进展。

与「单会话、单助手」相比，蜂群更适合**可分解、可并行、多角色**的任务，例如：调研 + 开发 + 评审、多模块并行开发、多步骤运维处置等。用户不必等整棵树跑完，可以在任意时刻介入任意子 Agent，实现真正的人机协同。

---

## 二、核心优势

### 2.1 任意动态创建 Sub-Agent

- 任务需要时，**任意成员**都可以继续向下拆分子 Agent，形成多级协作树，而不是只能拆一层。
- 拆分由助手根据任务自主判断，系统通过策略与配额防止无意义、无限扩张。
- 当用户明确要求「创建 N 个子 Agent」时，系统会约束为**恰好 N 个**，避免多建或重复创建。

### 2.2 可向任意 Agent 发送消息

- 不仅可以和根助手对话，也可以**单独给任意子 Agent 发消息**。
- 适合追问进度、补充约束、纠正方向，或把子任务直接交给某个成员执行。
- 根协调者、中间协调者、叶子执行者均可被独立指挥。

### 2.3 微信式聊天界面，随时介入任何子代理

- **中间主区域**采用类微信的气泡对话：用户消息与助手回复左右区分，助手内容支持 Markdown，并支持**边生成边显示**。
- **左侧**为房间列表与成员树，可折叠层级，并展示成员状态（空闲 / 执行中 / 失败）。
- **右侧**展示当前选中成员的补充信息（如流式输出、工具调用等）；没有内容时不显示空白面板，界面更干净。
- 用户随时选中某个成员即可对话，**不必等待整棵协作树结束**。

### 2.4 流式 Graph 动态展现协作状态

- **协作拓扑图**以树状结构展示房间内所有成员及上下级关系，支持缩放、拖拽浏览。
- 成员状态、消息与任务进展会**实时更新**，无需手动刷新页面。
- 执行过程中可一眼看出：谁在忙、谁已完成、谁出错，整体协作过程可视化。

### 2.5 零配置启动，开箱即用

- **无需预先创建数字员工或手工搭团队**，进入功能后一键「开始对话」即可自动创建房间与根 Assistant。
- 适合快速验证想法、临时复杂任务；需要专业角色时，再按需绑定数字员工或让 Assistant 自动拆分专长子 Agent。

### 2.6 可选绑定数字员工，兼顾「通用」与「专业」

- 子 Agent 既可以是**通用执行者**，也可以绑定 OpenOcta **数字员工**（垂直人设、精简技能与专属工具）。
- 同一房间内可混合多种角色，例如：根助手负责协调，下属分别为「运维专家」「开发助手」「评审助手」各干一块。
- 与 [数字员工](./digital-employees.md) 能力打通：专业能力与蜂群编排结合，而不是二选一。

### 2.7 成员之间可互相发消息，不仅限于用户指挥

- 不仅用户可以@某个成员，**成员之间也可以互相传递任务、追问或汇总结果**。
- 协调者可以把子任务分派给下属，下属完成后由上级或根助手汇总，更接近真实团队协作，而非「用户 ↔ 多个孤立机器人」。

### 2.8 列表 + 图谱双视图，兼顾操作与全局

- **成员树**：适合快速点选、查看层级与状态，日常发消息、介入的主入口。
- **协作拓扑图**：适合一眼看清整棵协作关系、谁挂在谁下面，适合汇报、复盘和复杂任务的全局把控。
- 两种视图联动：点树或点图均可切换当前对话对象，降低在大型协作树中「找不到人」的成本。

### 2.9 执行过程可观测，关键信息按需呈现

- 选中成员后，除主对话气泡外，还可查看**实时流式输出、工具调用**等补充信息（有内容才显示，避免空面板占位）。
- 便于判断某个子 Agent「正在想什么、调了什么工具」，方便用户决定要不要介入、何时介入。
- 助手回复支持 **Markdown 富文本**（标题、列表、代码块等），长回答更易读。

### 2.10 房间级管控，协作可收可放

- **全部停止**：一键终止当前房间内所有进行中的任务，避免失控跑量。
- **删除房间**：连同成员、协作记录、消息页会话一并清理，房间生命周期清晰，适合「一项目一房间」。
- **多房间并存**：可同时保留多个房间，分别对应不同项目或实验，互不干扰。

### 2.11 聊天记录双入口，蜂群与消息页互通

- 在蜂群里的每一次对话，都会沉淀到 **「消息」** 模块对应会话中。
- 适合两种习惯：在蜂群里「看着图协作」，在消息里「按会话检索、复制、回顾」。
- 删除房间时会同步清理，不会留下大量无用会话。

### 2.12 拆分数量可控，复杂任务不易「子 Agent 爆炸」

- 当用户明确要求「创建 N 个子 Agent / 孙子 Agent」时，系统会引导助手**按数量纪律执行**，并设有深度、人数上限，避免一次拆出过多角色。
- 在「能拆」和「拆得太多」之间取得平衡：该拆才拆，拆完能收。

### 2.13 子任务边界清晰，上下文不「一锅炖」

- 每个子 Agent 创建时会收到**独立、可执行的子任务说明**，专注自己的职责，而不是被动吞下整段用户对话。
- 协调者负责拆分与汇总，执行者负责落地，分工更接近真实项目里的「任务单」模式。
- 有利于长任务、多角色场景下**减少串线、减少无效上下文**，让每次模型调用更聚焦。

### 2.14 成员状态一眼可辨，出错可定点处理

- 成员树与拓扑图上会标注 **空闲 / 执行中 / 失败** 等状态，不必等最终文字才知道谁卡住了。
- 某个子 Agent 失败时，可**直接选中该成员**补充说明或纠正后重试，而不必推翻整间房间从头来。
- 配合「全部停止」，既能局部救火，也能在失控时一键收束。

### 2.15 全链路实时同步，协作过程「跟着动」

- 成员变化、消息流式输出、工具调用等会**自动推送到界面**，无需手动刷新页面。
- 左侧成员树、中间对话、右侧详情与拓扑图**联动更新**，适合长时间运行的复杂任务，随时掌握最新进展。

### 2.16 与 OpenOcta 主能力同栈，不是另起炉灶

- 蜂群成员复用平台统一的 **Agent 运行时、工具、技能（Skills）与数字员工** 体系，配置方式与日常单会话一致。
- 可在蜂群里绑定数字员工，也可让通用 Assistant 按需拆子 Agent，**编排与专业能力**可组合使用。
- 对话沉淀到「消息」模块，删除房间时一并清理，与平台其它模块的会话管理习惯一致。

### 2.17 协调规则可治理（团队可定制 SWARM 策略）

- 项目可提供 **SWARM 协调说明**（如团队统一的拆分原则、禁止事项、汇报格式），蜂群成员会自动遵循。
- 适合团队把「什么时候拆、拆几个、怎么协作」沉淀为规范，而不是每次靠口头叮嘱。
- 系统硬上限（深度、人数）与策略文案**双层约束**，兼顾灵活与可控。

### 2.18 基于 A2A 的协作底座，面向更大规模扩展

- 成员之间的任务派发与消息协作，建立在 **A2A（Agent-to-Agent）** 能力之上，为标准化、可互操作的 Agent 协作打基础。
- 当前以**单房间内协作树**为主；后续可演进为跨节点、跨实例的远程 Agent 混编（见第九章规划说明）。
- 对用户而言：今天获得的是「可视、可控的多 Agent 房间」；对产品演进而言：保留向**分布式蜂群**扩展的空间。

## 三、与单会话模式的对比（简要）

| 维度 | 单会话助手 | Agent Swarm |
|------|------------|-------------|
| 角色结构 | 单一对话线程 | 多成员树状协作 |
| 并行与分工 | 依赖模型在一轮里「扮演多角色」 | 可真实拆成多个子 Agent 分别执行 |
| 用户介入 | 只能回复当前会话 | 可介入任意成员 |
| 进度感知 | 主要靠文字滚动 | 成员树 + 实时拓扑图 + 状态标识 |
| 子任务上下文 | 同一会话内混合 | 按成员拆分 instruction，职责更清晰 |
| 失败处理 | 通常重开或继续同一线程 | 可定位到失败成员单独纠正 |
| 团队规范 | 依赖每次提示词 | 可配合项目级 SWARM 协调说明 |
| 适用任务 | 问答、单点修改、短流程 | 多步骤、多角色、可分解的复杂任务 |
| 资源消耗 | 相对较低 | 相对较高（见第九章性能说明） |

## 四、房间与协作规则

### 4.1 房间

- 蜂群以**房间**为协作单位，每个房间有独立成员树与对话历史。
- 每个房间默认有一位**根 Assistant** 作为入口，用户通常先向它描述任务。
- 支持新建房间、切换房间、删除房间；删除时会一并清理该房间内的成员、协作记录，以及在「消息」页中对应的聊天会话。

### 4.2 规模上限

为避免资源失控，系统对单个房间设有硬性上限：

| 限制项 | 上限 | 说明 |
|--------|------|------|
| 协作树深度 | **10 层** | 从根助手算起，最多 10 层 |
| 房间成员总数 | **55** | 含根助手及所有子 Agent |
| 单成员直接下属 | **5** | 同一成员下最多 5 个直接子 Agent |

达到任一上限后，将无法继续创建新成员，需由现有成员继续完成工作或由用户调整任务范围。

### 4.3 何时适合拆分

建议在以下情况才拆分子 Agent：

- 任务确实包含多个可并行或先后独立的子任务；
- 单一助手难以在合理步骤内完成，且拆分能带来明确收益；
- 每个子任务边界清晰，能用简短指令说清。

不建议为简单问答、单点修改、或「看起来更专业」而过度拆分。

## 五、与「消息」模块的关系

- 蜂群中每个成员的对话，会同步出现在 OpenOcta **「消息」**页的会话列表中。
- 用户既可在蜂群界面协作，也可在消息页按传统会话方式查看、回顾同一成员的完整聊天记录。
- 删除房间时，相关会话会一并删除，避免遗留无效会话。

## 六、界面与操作

### 6.1 入口

- 在顶栏进入 **AgentSwarm**（当前为 **BETA** 功能）。
- 首次使用可一键「开始对话」，系统自动创建房间与根助手。

### 6.2 主要区域

| 区域 | 作用 |
|------|------|
| 左侧 | 房间列表、成员树、成员状态 |
| 中间 | 与当前选中成员的微信式对话 |
| 右上 / 中部 | 协作拓扑图（Graph） |
| 右侧 | 选中成员的详情与实时输出（按需显示） |

### 6.3 常用操作

- **向根助手下达任务**：描述目标，可说明希望拆成几个子 Agent、各自职责。
- **选中成员发消息**：在成员树或拓扑图中点击成员，在中间栏输入并发送。
- **全部停止**：终止当前房间内所有进行中的任务。
- **删除房间**：结束协作并清理该房间全部数据（不可恢复）。

## 七、典型使用场景

1. **多角色协作**：例如「一人调研、一人写代码、一人做评审」，由根助手拆分后分别执行。
2. **多层级拆解**：子 Agent 在自身任务范围内继续向下拆分（在深度与人数上限内），例如「三个儿子各自再带三个孙子」类任务。
3. **中途介入**：发现某个子 Agent 方向偏差，直接选中它补充说明或纠正，无需重头开始。
4. **进度可视**：通过拓扑图观察整体推进，而不只依赖最终汇总结果。
5. **项目制房间**：一个客户需求或一次攻关对应一个房间，结束后删除，历史在消息页也可追溯（删除前）。
6. **专业 + 编排**：根助手负责拆任务，子成员绑定数字员工负责垂直执行（运维、开发、文档等）。
7. **协调者汇总**：由根 Assistant 收集各子 Agent 产出后，给用户一份统一结论。
8. **长任务盯盘**：任务运行较久时，通过实时拓扑与状态标识观察进展，在关键节点介入，而不只等最后一份总结。
9. **团队规范落地**：在 SWARM 协调说明中约定拆分上限、汇报格式，让不同成员行为更一致。
10. **局部失败重试**：某个子 Agent 报错后，直接对该成员补充约束或改派，无需重建整棵树。

## 八、典型使用流程

1. 打开 **AgentSwarm**，创建或选择一个房间。
2. 向根 Assistant 描述复杂任务（可明确子 Agent 数量与分工）。
3. 在拓扑图 / 成员树中观察子 Agent 的创建与状态变化。
4. 需要时选中某个子 Agent，在中间栏**直接对话**介入。
5. 如需回顾，可在 **「消息」** 页打开对应会话查看完整历史。
6. 任务结束使用 **「全部停止」**；不再需要该房间时 **删除房间**。

## 九、重要说明与使用建议

### 9.1 A2A 协议与后续扩展

- 蜂群成员之间的协作已基于 **A2A（Agent-to-Agent）** 能力构建，为跨 Agent 协作预留了标准接口。
- **后续规划**：开放对外 A2A 接入，使**其他 Agent 可远程、跨节点**调用本房间的成员，与其他 OpenOcta 实例或兼容 A2A 的 Agent 组成**更大规模的分布式集群**。
- 当前版本以**本机单房间内的协作树**为主；跨节点集群属于后续演进方向。

### 9.2 性能与 LLM Trace

> **注意：开启 Agent Swarm 后，部分节点回答可能变慢。**

蜂群会触发多轮、多成员的模型调用与工具执行，整体耗时通常高于单会话对话。

### 9.3 其他建议

- 删除房间**不可恢复**，删除前请确认不再需要该房间的成员与记录。
- 蜂群当前为 **BETA**，交互与配额可能随版本迭代调整。
- 复杂任务建议先向根助手说明期望的子 Agent 数量与分工，减少反复拆分。
